  1. 100% LoCoMo — retired. Honest numbers from the same file: 60.3% R@10 raw, 88.9% R@10 hybrid v5.
  2. 100% LongMemEval — retired. Hand-coded against three dev-set questions (quoted-phrase boost, person-name boost, "I still remember" pattern). Acknowledged as test contamination.
  3. recall_any@5 as headline metric — retired. Contributor @gizmax built qa_bench_v2.py to run the real QA pipeline with a calibrated GPT-4o judge.
  4. "2× Mem0" — retired as a metric mismatch (retrieval-recall vs end-to-end QA).
  5. 1,986 LoCoMo question count — acknowledged as the inflated count that includes the 446 adversarial subset. Future reports will use the 1,540 non-adversarial subset with labelling.
  6. "100% beats every product, no API key, no cloud" — acknowledged as a dishonest weld of two different configurations. Fixed in commit aa10f8f.
  7. Harness bug #242 (assistant-turn discard) — fixed first, before any public re-benchmark.

The honest end-to-end QA numbers on the corrected qa_bench_v2.py harness are:

0.668 raw  ·  0.672 rooms  ·  0.532 aaak

Those are the numbers that belong in a comparison table alongside Mem0, Zep, and Mastra. The recall_any@5 headline is retired.

Competitive intelligence · MemPalace / Milla Jovovich

MemPalace's "100% on LongMemEval" is a metric category error, not a breakthrough.

Verdict: The claim is real in the sense that the project and numbers exist, but methodologically suspect and not comparable to the published LongMemEval leaderboard. MemPalace's benchmark runner measures session-level recall_any@5 retrieval — not end-to-end QA with a judge — and the specific 100% number was hand-patched against three failing questions on the dev set. The honest raw number (96.6% R@5 with default embeddings, no LLM) is a respectable retrieval result for a zero-cost local system, but it is not a LongMemEval score in the sense the published leaderboard means, and does not invalidate Nominex's partial-context thesis.
1 · What we found

Primary-source evidence

The project exists and is real. On 5 April 2026, a GitHub repository at github.com/milla-jovovich/mempalace was published under an MIT licence by an account registered to Milla Jovovich (the actress — Resident Evil, The Fifth Element). The engineering was done by Ben Sigman, a developer and crypto-adjacent founder; Jovovich is credited as "architect" and public face. The pair used Anthropic's Claude Code to build the system. Within 48 hours the repo passed 23,000 stars and briefly became the #1 trending repo on GitHub, driven by mainstream tech press coverage (Decrypt, RT, Yahoo, Bitcoin.com, Kotaku) that leaned on the celebrity angle.

The product name is MemPalace (the inspiration is the ancient method-of-loci mnemonic technique). Product home: mempalace.tech.

The 100% claim originated in Ben Sigman's launch tweet, which stated verbatim: "100% recall on LongMemEval — first perfect score ever recorded. 500/500 questions. Every question type at 100%." A screenshot of the tweet is embedded in GitHub Issue #29, which we treat as the canonical critique.

The critique is also primary-source and devastating. Issue #29, titled "Multiple issues with benchmark methodology and scoring", was filed on 7 April 2026 by GitHub user dial481, who discloses they work on a different AI memory project and are the author of a public LoCoMo ground-truth audit (github.com/dial481/locomo-audit). The issue was closed by the milla-jovovich account on 9 April 2026 with 41 thumbs-up reactions. Crucially, most of the critique is sourced to MemPalace's own benchmarks/BENCHMARKS.md file, which openly documents the methodological caveats that the launch post stripped out.

Independent coverage of the controversy is available at Cybernews, Penfield Labs Substack ("None of the benchmark scores are real"), Nicholas Rhodes' review, and danilchenko.dev. All four converge on the same conclusion: real project, legitimate raw-retrieval number, misrepresented as a LongMemEval leaderboard score.

2 · The architecture, as best we know

How MemPalace actually works

Architecture class: local-first vector DB + hierarchical metadata layer, closer to a ChromaDB skin than to a knowledge graph or agentic memory framework.

  • Storage. SQLite for structured state, ChromaDB for vectors. Runs entirely on the user's machine. Zero cloud dependencies in "raw" mode.
  • Embeddings. Default ChromaDB embeddings (all-MiniLM-L6-v2, 384-dim) — an old, small, open-weight encoder, not a frontier model.
  • Hierarchy. A fixed vocabulary of metadata categories the author calls wings / halls / rooms / closets / drawers / tunnels. In practice this is metadata filtering on top of ChromaDB — a standard Chroma feature, not a novel retrieval mechanism. Critics (including the Issue #29 author) call out the "+34% palace gain" as a comparison between unfiltered and metadata-filtered search, not evidence of architectural innovation.
  • Write-time compute. On ingest, conversation data is stored verbatim (not summarised), then classified into wings/halls. The "AAAK" dialect is a claimed 30× lossless compression format readable by any LLM without a decoder — but no round-trip information-retention eval ships with the repo, and applying AAAK drops the retrieval score from 96.6% to 84.2% (12.4 points), which contradicts the "lossless" framing. Realistic description: structured shorthand, not lossless compression.
  • Retrieval. Two-pass: classify the query into a hall, then do cosine-distance search weighted by hall match. Top-k results returned. In the "hybrid" mode that produces the 100% number, results are then reranked by a paid Claude Haiku call.
  • Base model. For the raw 96.6% result, no LLM at all — just embeddings and cosine distance. For the 100% hybrid-mode result, Claude Haiku for reranking on LongMemEval (and Claude Sonnet for the separate LoCoMo number). So the "no API key, no cloud" marketing describes the raw mode, and the "100%" marketing describes the hybrid mode — two different configurations fused in the launch copy.
  • Semantic vs episodic. No explicit separation. Everything goes in as verbatim session text, organised by metadata category.
  • Agentic workflow. No planner, no reflection, no summariser. It's a retrieval system.

Net: MemPalace is a well-packaged local RAG system with an old embedding model, a metadata taxonomy, and an optional Haiku reranker. It is not a new memory architecture in the sense that Letta, MemGPT, Zep, or PMM are. The honest technical contribution is "here's a zero-cost local retrieval setup that works surprisingly well on user-turn-only embeddings."

3 · The claim, parsed

What the 100% actually measures

This is the crux. Understanding it requires separating three things that MemPalace's launch post conflated: (a) which benchmark, (b) which metric, (c) which scoring model.

Which benchmark

LongMemEval-s, the "small" variant of LongMemEval. 500 questions, haystacks of roughly 50 prior sessions per question, around 115K tokens per haystack. Full benchmark, not a sub-category. No modification to the question set itself.

Which metric (this is the category error)

LongMemEval as published is an end-to-end QA benchmark. A system must (1) retrieve relevant sessions from the haystack, (2) generate an answer, and (3) have that answer marked correct by a GPT-4 judge. Published leaderboard numbers are the percentage of questions where the generated answer was judged correct.

MemPalace's benchmarks/longmemeval_bench.py does step (1) only. It never generates an answer. It never invokes a judge. For each question it:

  1. Concatenates only the user turns of each session (assistant turns are not indexed at all).
  2. Embeds with the default ChromaDB MiniLM encoder.
  3. Returns the top 5 sessions by cosine distance to the question text.
  4. Reads the gold answer_session_ids labels that ship with the dataset, and if any one of them appears in the top-5, awards a score of 1.0.

This is recall_any@5. Not LongMemEval. A system that retrieves the right session and then answers wrong scores 100% on this metric and 0% on the published leaderboard. An independent tester (parcadei on X, cited by Penfield Labs) plugged MemPalace into a real end-to-end QA harness and reported that when you actually have an LLM answer questions using MemPalace's retrieved context, the right answer comes back roughly 17% of the time. That number is unverified by us, but it's directionally consistent with what the runner pipeline does vs. what it claims.

To add insult: the runner also computes recall_all@5 (all gold sessions must appear in the top-5), and chose to report recall_any. On multi-gold-session questions recall_any is substantially softer.

Which scoring model

There is no GPT-4 judge in MemPalace's pipeline at all. The "100% with Haiku" number uses Claude Haiku as a reranker, not as a judge. Haiku looks at the top candidates and reorders them; scoring is still label-set overlap, not answer correctness. So the "100%" reflects "after Haiku rerank, a gold session ID is in the top-5 100% of the time" — on a test set whose failing cases were patched by hand (see next section).

Which base model for generation

There isn't one. MemPalace doesn't generate answers. Every frontier-model concern about benchmark contamination is therefore moot for this specific claim — because the claim isn't about question answering, it's about retrieval recall on user-turn embeddings. This is both the generous interpretation ("no, they didn't cheat with a bigger model") and the damning one ("the metric is not what you think it is").

4 · Credibility assessment

Does the claim survive scrutiny?

Our honest per-concern read. Everything here is sourced to Issue #29, which is itself sourced to MemPalace's own BENCHMARKS.md:

ConcernRatingEvidence
Overfitting / teaching to the test Likely concern MemPalace's own BENCHMARKS.md states: "This is teaching to the test. The fixes were designed around the exact failure cases, not discovered by analyzing general failure patterns." The 100% was reached by adding three targeted patches (quoted-phrase boost, person-name boost for "Rachel", high-school-reunion pattern match) against three specific dev-set failures. The held-out 450 number is 98.4%.
Non-standard scoring (category error) Verified concern Runner computes recall_any@5 retrieval, not end-to-end QA. No answer generation, no GPT-4 judge. All three numbers (96.6, 98.4, 100) are retrieval numbers dressed as LongMemEval scores. The honest framing would be "MiniLM recall on user-turn-only embeddings," which is not a published benchmark.
Rubric gaming (recall_any vs recall_all) Likely concern Runner computes both recall_any@5 and recall_all@5 and reports the easier one. For multi-session questions, this roughly multiplies the odds of credit by the number of required sessions.
Answer leakage Unclear / N/A No answer generation pipeline, so no classical leakage surface. However, gold session IDs are read directly from the dataset to score retrieval — trivially true by construction.
Single-run variance Not a concern for raw 96.6% Retrieval with fixed embeddings is deterministic. The raw 96.6% number is reproducible and dial481 (the critic) confirms so directly.
Benchmark contamination of base model Not applicable No generative model in the raw pipeline. Haiku's role is reranking a small candidate set, not answering from training-data memorisation.
Local/free deployment marketing Likely concern The 100% requires a paid Claude Haiku call. The "no API key, no cloud" description belongs to the raw 96.6% mode. Two configurations welded together in marketing copy.
Related claims (AAAK "lossless", LoCoMo 100%) Multiple concerns LoCoMo 100% is from top_k=50 on conversations containing 19–32 sessions — retrieval is structurally bypassed because every candidate is always in the pool. AAAK has no round-trip eval and reduces LongMemEval score by 12.4 points, which is not lossless by any definition.

Plain-English summary. The "100% on LongMemEval" headline fails in two independent ways: first, the runner measures the wrong thing (retrieval recall, not end-to-end answer correctness); second, even granting the wrong thing, the 100% required three hand-coded patches targeted at three specific failing questions. The raw, unpatched, no-LLM number on the same runner is 96.6% — which is a genuine and reproducible retrieval result for a zero-cost local setup, but it is not comparable to any number on the published LongMemEval leaderboard and should not be cited as one.

5 · Strategic implications for Nominex

What this means for our positioning

  • MemPalace does not measure the surface Nominex argues about. Our partial-context trap thesis is about what happens when an agent generates and reasons under a lossy, retrieval-shaped window into prior state. MemPalace's headline number is a retrieval-only metric that never gets to the reasoning step. The two systems are measuring different stages of the same pipeline. Their "100%" is silent on our thesis — it doesn't support it and it doesn't refute it.
  • Nominex is not in direct competition. MemPalace is a local single-agent RAG tool with a fixed metadata taxonomy. Nominex / PMM operates cross-agent, cross-provider, cross-tool with async enrichment and a structured semantic layer. The architectural distance is large enough that they are adjacent, not competing. Their success actually widens the "structured memory matters" tent we want to pitch under.
  • The v1.5 related-work section should cite this carefully. We should reference MemPalace as an example of the framing confusion in the space — a project whose own BENCHMARKS.md candidly documents that its 100% is teaching-to-the-test, but whose launch copy stripped every caveat. This is a useful foil for the "what does a real memory eval look like?" argument. Do not dismiss it; credit the honest 96.6% raw-retrieval result and then separate it cleanly from LongMemEval leaderboard numbers.
  • The reframe is that MemPalace supports our argument, not undermines it. The controversy demonstrates (a) the market is hungry for local structured memory — 23K GitHub stars in 48 hours validates demand, (b) existing memory evals are easy to game and misreport, which is exactly the gap a more honest benchmarking effort could fill, and (c) celebrity hype outperforms rigour in short-term attention, which is a marketing lesson we can absorb without copying.
  • The bear case if we're wrong. If an independent party reproduces a high-90s number on a properly-judged LongMemEval run using MemPalace's retrieval + any modern answerer, the headline would still be compromised on process, but the underlying claim ("hierarchical local retrieval is competitive with paid cloud memory products") would get stronger. The bear case for Nominex is that "good enough local retrieval" becomes the default and the cross-agent story has to do more work. We should pre-empt this by making the cross-agent / cross-provider value explicit, not just the quality-of-memory value.
6 · Open questions

What we don't yet know

  • Does a proper end-to-end run of MemPalace + any frontier answerer actually hit the high 90s on real LongMemEval QA? The 17% figure from parcadei is one data point, unverified and not cited to a reproducible harness. This is the single most important number we don't have.
  • Will the maintainers correct the launch messaging, or just the README? As of the issue being closed on 9 April, headline numbers on mempalace.tech still feature the 100% framing with a footnote. Worth re-checking in a week.
  • What is Mila / Milla's actual technical involvement? Reporting uniformly says Ben Sigman did the engineering and she is the "architect" and public face. We have no evidence of her prior AI / memory / benchmarks work and believe the credibility weight sits entirely on Sigman and any future third-party reproductions.
  • Has the LongMemEval author team (Xiaowu et al.) commented publicly? A statement from the benchmark maintainers disclaiming the "first perfect score" phrasing would be the clearest resolution. We did not find one; worth a targeted search in a few days.
  • Is there any independent reimplementation of MemPalace's hierarchy on a different embedding model? Would separate the "hierarchy helps" claim from the "MiniLM is surprisingly OK on LongMemEval-s" claim. Not found in our search.
  • Does the LongMemEval leaderboard have or gain a "Memory Palace" entry? We did not verify the upstream leaderboard directly in this pass — flagged for follow-up.
